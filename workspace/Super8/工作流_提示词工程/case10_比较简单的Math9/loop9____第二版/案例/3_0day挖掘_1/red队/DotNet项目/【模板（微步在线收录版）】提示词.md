# 我的重要关注点

现在，我需要进行系统的代码安全审查，但这一次的目标更偏向于【微步在线收录口径】。重点关注于：

- 关注的【漏洞风险等级】：危急、高危、中危。
- 第一优先级：
    - 【无权限要求的前台漏洞】
    - 【通用漏洞】（这里的“通用”指：产品官方源码、官方默认配置、官方文档、官方交付物中自带的问题；只要是官方仓库/标准发布内容里带出来的，就优先按“通用型”理解。只有明确属于某个客户自己的误操作、客户私有环境事故、客户侧弱口令处理、客户自己的文档泄露、客户私有二开失误时，才按事件型理解）
    - 【具有可利用 Exploit / 可直接复现的利用链】
- 第二优先级：
    - 前台可达、前置条件较少、能较清晰形成利用链的漏洞。
- 默认降级但仍可保留记录：
    - 需要登录/认证的漏洞
    - 后台/管理员面漏洞
    - 事件型漏洞
    - 依赖很强环境假设、只能证明“存在可能性”的漏洞
- 关于漏洞类型：
    - 继续重点关注【文件上传、RCE命令执行/反序列化、SQL注入、组件漏洞】这类更容易满足“可利用、通用、可收录”要求的问题。
    - 对【XSS、CSRF、IDOR】不再一刀切降级：
        - 如果它们满足【无权限前台 + 通用 + exploit清晰】的条件，可以保留并提高优先级。
        - 如果它们仍然依赖组合利用、依赖认证/管理员、或依赖较强前提条件，则降低优先级。
- 输出时，不能只说“可能有风险”或“存在可能性”：
    - 要尽量判断是否真的具备 exploit 价值
    - 要明确它是【前台/后台】、【无权限/有权限】、【通用/事件型】
    - 要明确它更适合归入：
        - 【微步在线收录候选】
        - 【后台重大漏洞（酌情池）】
        - 【仅内部保留】
- 仍然不需要冗余性的常识讲解，比如：
    - 不需要【漏洞常识层面的原理】
    - 不需要大段泛泛危害描述
    - 不需要教科书式成因展开
