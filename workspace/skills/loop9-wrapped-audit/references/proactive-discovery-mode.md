# 主动目标发现模式（weibu-submission）

## 目的

自动 audit 线在 `weibu-submission` 模式下，目标不是消费本地已有 `targets/` 池，
而是：

1. 汇总候选来源
2. 过滤本地已处理/已在跑目标
3. 只在符合微步实际潜规则的候选里选出目标
4. 下载到本地
5. 再调用标准 wrapper 发起审计

## 当前业务主轴（2026-03-25 起进一步收紧）

`weibu-submission` 默认不再理解成“泛海外 common web app 挖掘”。

应优先理解为：
- **中国大陆正在使用** 的 CMS / OA / 常见 WEB 应用 / 相关通用软件
- 默认优先有明确厂商/公司归属的项目
- 海外项目只保留窄例外通道

## 候选来源优先级

默认按这个顺序考虑：

1. 微步已列好的项目/范围
2. `gitee` 上的厂商/公司项目
3. `gitee` 上的高影响个人项目
4. `gitcode` 上的厂商/公司项目（注意镜像风险）
5. `gitcode` 上的高影响个人项目（注意镜像风险）
6. `github` 上的**窄例外通道**项目

## GitHub 例外通道

`github` 候选默认不是主战场，只在以下情况进入候选池：

- 微步范围/名单中明确给出名称的
- 类似 Apache / Shiro 这类会被大量中国国内产品/框架使用的通用组件
- 虽属海外项目，但中国大陆政府/国企/大厂实际使用很多的项目

## 当前准入门槛

候选在进入最终排序前，必须先全部满足：

- 符合 **中国大陆使用导向**；若不符合，则必须明确落入 GitHub/海外例外通道
- `ownership_kind=vendor|company` 优先；若为个人项目，则只能在影响面明显足够广时保留
- `stars` 门槛：
  - 公司/厂商项目：`>= 1000`
  - 个人项目：`>= 1500`
- 非本地已处理 / 已在跑
- 非明显靶场 / deliberately vulnerable app
- 对支持该字段的平台：非 archived、非 fork
- 对 GitHub：`pushedAt >= 当前执行时间 - 6个月`
- 对 Gitee / GitCode：页面可见的代码维护信号应显示近 6 个月仍有有效维护活动
- 对 GitCode：若看起来只是从 GitHub 镜像过来，默认降级或淘汰，除非它本身就是该业务范围明确要收的目标

## 当前排序直觉（仅对已过硬门槛的候选生效）

默认更偏好：

1. `weibu-listed`
2. `gitee-vendor`
3. `gitee-user`
4. `gitcode-vendor`
5. `gitcode-user`
6. `github-exception`

并且更偏好：
- 有明确厂商名 / 官网 / 联系域名 / 官方组织
- 更像中国大陆常见业务系统，而不是泛社区玩具或偏海外 SaaS 语境
- 更适合当前审计性价比：
  - 不是 star 越大越优先
  - 在同等口径下，优先更容易产出有效漏洞、代码质量未必最硬核的目标

## 资产证据增强层（默认价值门槛层）

`loop9-asset-evidence` 现在不应再只被理解成“少量候选的比较层”。

在 `weibu-submission` 里，它更应被理解为：
- **默认必经的价值门槛层**
- 用来判断一个 repo-side 看起来可做的候选，是否已经达到当前发车价值

推荐使用方式：
- 当一个候选在 repo 侧已经看起来“像能做”时，就尽快过这层门槛
- 如果它没有达到当前发车价值，就不应继续被当成正式目标长期养着
- 如果它已经达到发车下限，就可以直接成为本轮候选，不必为了比较而强行再养出 1~3 个陪跑对象
- 只有当多个候选都已经过门槛且仍然模糊时，这层才兼任“薄比较层”

这层的定位必须保持克制：
- 不是新的 discovery brain
- 不是大规模资产导出器
- 不是最终发车器
- 它首先是价值门槛层，其次才是少量候选比较层

当前应记住三类样本：
- `ZLMediaKit`：强样本 / 高优先级国内实战型
- `UReport / UReport2`：有部署但当前高价值信号仍偏弱的对照样本
- `MaxKey`：title clue 主识别面 / 中等规模 / 高价值场景提前出现

当前用户口径下的薄使用规则：
- 理想目标：中价值及以上
- 当前发车下限：允许存在一个“勉强够格可发车”层
- `1Panel` / `MaxKey` 当前更接近这一层
- `UReport / UReport2` 当前仍未被拉升到这一层
- 这层输出的是保守价值判断，不替代最终人工/AI拍板

## 与“双槽位 audit dispatcher”的关系

这里要明确区分两层：

1. **单轮 discovery**
   - 仍然只选 **1 个** 仓库
2. **audit dispatcher 外层**
   - 如果当前 audit 槽位未满，可以顺序执行多轮单目标 discovery
   - 直到补满目标槽位或没有合格候选

也就是说：
- 这里不把 discovery 本身改成双选器
- 而是把“补位到 2 个槽位”放在 dispatcher 外层完成

## 重要边界

- 单轮 discovery 只选 **1 个** 仓库
- 选不出合适目标时，宁可返回“本轮无合格候选”，也不要硬凑
- discovery 的模糊判断不要下沉成新的本地 Python brain
