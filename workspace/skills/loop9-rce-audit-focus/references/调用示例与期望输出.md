# 示例 1：早期轻量预扫

## 适用场景
- repo 已经本地化
- 主审计已经对结构有初步理解
- 但尚未锁死到某个单点漏洞
- 现在希望从 **RCE / 文件上传 / 弱鉴权推进到执行面** 的视角做一次专题预扫

## 示例调用意图
> 请使用 `loop9-rce-audit-focus`，对当前项目做一次轻量 RCE 专题预扫。目标不是证明漏洞，而是识别高价值 exploitation surface，并初始化 `RCE专题共享上下文.md`。

## 好的执行重点
- 从功能面切入，不从危险函数 grep 起手
- 优先看：
  - 上传 / 导入导出
  - 模板 / 主题 / preview / render / compile
  - webhook / test connection / scheduler / plugin / package
  - JWT / SSO / auth-provider / user-source / config
  - 中间件 / 管理面 / 控制台
- 给出 1~3 个最值得后续深挖的方向
- 初始化 Markdown shared context

## 好的输出应该像什么

### 不是这样
- “命中 Runtime.exec、system、eval 共 12 处，建议继续审计。”
- “发现上传功能，可能 RCE。”

### 更像这样
- 当前项目最值得关注的功能面有三块：
  1. 后台主题模板编辑与 preview 渲染链
  2. 附件上传与媒体处理链
  3. SSO/JWT 用户源配置与后台集成能力
- 从现阶段看，它更像“低权限后台推进到执行面”的项目，而不是纯直达命令执行型。
- 已初始化 `RCE专题共享上下文.md`，记录了高价值功能面、角色边界、上传链观察和两条初步成链方向。
- 当前与微步偏好的贴合度：**中等偏上**。还没到可直接定 RCE，但已经进入值得持续关注的路径。

## 这一轮 shared context 的好样子
应该至少有：
- 为什么纳入 RCE 专题
- 高价值功能面
- 当前角色边界观察
- 1~3 条成链方向的雏形

不需要：
- 把所有目录和所有函数名都抄进去
- 写成 JSON 样式字段表

# 示例 2：强信号出现后的专题深挖

## 适用场景
主审计已经发现明确强信号，例如：
- 低权限后台可编辑邮件模板
- 上传点落盘路径可预测
- test connection 可控目标
- 自注册用户可访问用户源/JWT 配置接口
- preview/render 链能接触用户输入

## 示例调用意图
> 当前主审计已经发现：低权限后台用户可编辑邮件模板，并存在 preview/render 能力。请使用 `loop9-rce-audit-focus` 围绕这一信号做专题深挖，重点判断是否能形成“低权限模板编辑 → 模板执行 → RCE”或相邻链路，并增量更新 `RCE专题共享上下文.md`。

## 好的执行重点
- 不要回头泛扫全仓
- 紧扣当前强信号
- 判断：
  - 这是执行面还是推进面？
  - 需要什么前提？
  - 当前已掌握的证据够不够？
  - 下一步最值钱的验证是什么？
- 允许投入较多时间和 token，只要线索质量够高

## 好的输出应该像什么

### 不是这样
- “模板功能存在，可能 SSTI，建议继续测试。”

### 更像这样
- 当前强信号已经足够把这条线升级为优先深挖对象。
- 现阶段更像的链路是：
  - **链路 A：低权限模板编辑 → preview/render 调用 → 模板表达式执行 → RCE**
  - **链路 B：低权限模板编辑 → 路径控制/包含链 → 本地模板文件解释执行**
- 当前证据：
  - 低权限角色可进入模板编辑页
  - preview/render 明确存在
  - 模板上下文包含可调用对象/辅助函数痕迹
- 当前缺口：
  - 尚未确认模板引擎是否允许表达式执行
  - 尚未确认 preview 路径是否与正式渲染共用同一执行栈
- 建议下一步：
  1. 先确认模板引擎类型与表达式能力
  2. 再确认 preview/test 是否可达真实渲染执行链
  3. 若成立，再评估是否贴近微步偏好的 RCE 收敛方向
- 已对 `RCE专题共享上下文.md` 做增量更新，并补充两条成链假设与当前证据/缺口。

# 示例 3：上传线的高质量分析输出

## 常见坏输出
- “有上传点，可能 getshell。”

## 更好的输出模式
- 上传入口位于后台素材管理模块，普通编辑角色可达。
- 前端限制为 jpg/png/gif，但后端实际接收逻辑依赖请求参数中的扩展名与 MIME，内容检测尚未看到。
- 落盘位置位于可预测路径，且文件名规则可控度较高。
- 当前更值得关注的不是“直接传 php shell”，而是：
  1. 是否存在替代扩展名 / 双扩展名
  2. 是否能通过路径遍历影响最终落点
  3. 是否有后续 render/include/thumbnail/preview 链会解释处理该文件
- 当前判断：这是一个**高价值上传推进面**，但暂时还不能直接写成 RCE。

# 示例 4：与微步偏好的贴合度判断怎么写

## 不好的写法
- “感觉挺危险，应该符合微步方向。”

## 更好的写法
- 当前链路和微步偏好的贴合点在于：
  - 它不是普通业务逻辑缺陷，而是在向 **RCE / 文件上传 + 权限问题 / 低权限后台推进高危执行** 收敛。
- 当前仍不够贴的地方在于：
  - 还缺少“执行面真正可达”的关键证据。
- 因此当前判断为：
  - **值得继续投入，但暂不拔高为已成立 RCE。**

# 示例 5：shared context 的“增量更新”应该怎么写

## 不好的更新方式
- 每次都重写整篇
- 每次都把所有历史观察重复一遍
- 只追加原始日志，不做提炼

## 更好的更新方式
在文末新增一个小节，例如：

```md
# 10. 本轮增量更新摘要

- 新增观察：低权限编辑角色已确认可访问模板预览功能
- 更新判断：模板线从“可疑功能面”升级为“高优先级成链方向”
- 新增链路：低权限模板编辑 → preview 渲染 → 模板表达式执行
- 对后续人的提示：先确认模板引擎与 preview 调用栈，不要急着回头泛扫上传面
```

这样后续人一眼就知道这轮到底推进了什么。

# 示例 6：什么时候值得大量投入

以下情况通常值得在该 skill 上投入较多时间和 token：
- 低权限角色已经接近模板/配置/执行能力
- 上传链不仅存在，而且落点/解释路径开始清晰
- 自注册/弱鉴权与后台高价值能力发生连接
- 中间件/控制台暴露，且配置修改明显能推进到脚本/模板执行

## 高投入时的正确心态
- 可以深
- 但不能散
- 重点不是“多扫点”，而是“把链压实”
