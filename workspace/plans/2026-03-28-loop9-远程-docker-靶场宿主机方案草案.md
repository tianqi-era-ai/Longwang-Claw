# Loop9 远程 Docker 靶场宿主机方案草案（v0）

## 文档定位

这不是“把 Loop9 整体搬上云”的方案。

这是一份 **“MacMini 本地控制面 + 腾讯云远程 Docker 靶场宿主机”** 的第一版架构与实施草案。

当前明确前提：

- **MacMini 本地继续承担**：OpenClaw、dispatcher、cron、audit、real-poc、opencode、计划/状态/报告主流程
- **腾讯云服务器只承担**：Docker 靶场宿主机 + 公网复测承载机
- 最终目标是：当某个 Loop9 项目需要对外提供复测环境时，由 MacMini 远程驱动云机拉起 Docker 环境、开放公网端口，并把可复测信息交付给审核人员

---

## 一、当前已确认的业务边界（source of truth）

### 1. 云机角色

腾讯云机器 **只扮演 Docker 靶场宿主机**，不承担 Loop9 控制面。

### 2. 控制模型

第一版保持极薄：

1. MacMini 是唯一控制面
2. 云机只接受 SSH 驱动
3. 所有“起环境 / 关环境 / 查状态 / 清理”的动作都走 SSH 远程命令

### 3. 公网开放对象

- 对外暴露的是 **按需启动的 Docker 靶场环境**
- 不暴露 OpenClaw 主控、不暴露 dispatcher / cron / Loop9 主流程
- 目的面向微步情报社区审核人员公网复测

### 4. 暴露方式

第一版接受：

- **直接开放容器端口**
- 不先引入复杂统一网关平台
- 但端口分配必须可追踪，且默认不开、按需开

### 5. 安全边界

- 主控与高风险暴露面已经物理分离：
  - MacMini = 控制面
  - 腾讯云 = 靶场面
- 因此本项目的主要风险不再是“OpenClaw 主控被公网拖下水”，而是：
  - 靶场复测现场被污染
  - 端口/环境失控
  - 云机逐渐雪花化

### 6. 资源现实

- 云机配置：2C / 8G RAM / 150G Disk
- RAM 8G 不可变
- 第一版接受“小规模、低并发、可重建”的设计原则

---

## 二、第一版总体设计目标

第一版的目标不是“把所有事情自动化到极致”，而是：

1. 让 MacMini 能 **稳定远程驱动** 云机上的 Docker 靶场
2. 让每个靶场具备 **明确生命周期**
3. 让公网暴露具备 **明确开关与追踪**
4. 让复测交付具备 **统一模板**
5. 让云机保持 **可清理、可回收、可重建**

第一版刻意 **不追求**：

- 不追求多租户平台
- 不追求复杂 Web 面板
- 不追求大量环境并发常驻
- 不追求“一次性做完整资产管理系统”

---

## 三、第一版设计原则

### 原则 1：控制面与靶场面严格分离

- 所有调度、判断、计划、报告，留在 MacMini
- 云机只做“被控制的 Docker 宿主机”
- 云机上不新增与 Loop9 主流程耦合过深的复杂逻辑

### 原则 2：默认按需启动，默认按需暴露

- 环境可以提前部署，但默认不开放公网
- 需要复测时才进入开放窗口
- 复测结束后，应尽快关闭暴露、决定保留或销毁

### 原则 3：现场可控优先于绝对安全洁癖

此项目不是“生产业务系统上线”，而是“高风险靶场复测承载”。

因此目标不是绝对防御，而是：

- 在复测窗口内保持环境稳定
- 必要时能对白名单控流
- 被污染后可快速重置

### 原则 4：可重建优先于手工养护

- 云机不应该被养成越来越脏的“雪花机”
- 任何长期有价值的信息，应尽量保存在：
  - MacMini 侧状态
  - 版本控制文档
  - 可重放脚本
  - 环境配置模板
- 云机上的价值应更多是“可快速重建的运行态”而不是“手工修出来的历史现场”

### 原则 5：先单靶场闭环，再扩展多靶场规则

第一版先验证单靶场闭环：

1. 远程起环境
2. 开公网
3. 复测通过
4. 关闭回收
5. 必要时重建

只有单闭环跑顺，再扩展到多靶场并发、端口池、白名单切换、自动清理。

## 四、目标架构（v0）

## 4.1 角色划分

### A. MacMini（控制面）

职责：

- Loop9 主流程
- 审计与 real-poc 主线
- 计划 / 状态 / artifacts / 报告
- 远程 SSH 控制云机
- 生成“靶场交付信息”

### B. 腾讯云服务器（靶场宿主机）

职责：

- 存放工作副本 / compose / volume / 容器
- 拉起 Docker 靶场
- 按规则开放端口
- 提供公网可访问环境
- 承担复测窗口内的运行面

### C. 审核人员（公网访问方）

通过：

- IP:Port
- PoC 说明
- 测试账号（如有）
- 时间窗口说明

进行复测。

---

## 4.2 数据与控制流

### 控制流

1. MacMini 选择某个项目需要上云复测
2. MacMini 通过 SSH 将必要的环境材料同步或在云机上生成工作副本
3. MacMini 远程执行 `docker compose up -d` 或等效命令
4. MacMini 远程检查环境健康状态
5. MacMini 为该靶场分配或确认对外端口
6. MacMini 控制腾讯云安全组 / 宿主机访问策略（默认关、按需开）
7. MacMini 生成交付信息并发给审核人员
8. 复测结束后，MacMini 关闭暴露并决定：保留 / 快照 / 销毁

### 数据流

- **长期状态**：优先保存在 MacMini
- **运行态数据**：保存在云机
- **交付信息**：由 MacMini 汇总并对外发送

## 五、第一版云机目录与资源组织建议

建议在云机上统一使用一个根目录，例如：

```text
/opt/loop9-range/
  hosts/
  environments/
  compose/
  volumes/
  logs/
  runtime/
  exports/
  tmp/
```

建议职责如下：

### `/opt/loop9-range/environments/<env_id>/`

单个靶场的工作目录，建议包含：

```text
<env_id>/
  source/                # 工作副本源码（如需要）
  compose/               # 运行时 compose 文件 / override 文件
  env/                   # .env / secrets 占位 / 运行配置
  state/                 # 本环境状态文件
  logs/                  # 环境级日志索引
  snapshots/             # 可选的环境级快照元信息
  notes/                 # 本环境特殊说明
```

### `/opt/loop9-range/runtime/`

全局运行时数据，例如：

- 端口租约表
- 当前活跃环境索引
- 当前开放窗口索引
- 资源统计缓存

### `/opt/loop9-range/logs/`

宿主机级操作日志：

- 起环境
- 关环境
- 开放端口
- 关闭端口
- 清理操作

### `/opt/loop9-range/exports/`

可选的导出目录：

- DB dump
- 初始化快照导出
- 审核前临时留存文件

## 六、靶场状态机（第一版强约束）

每个靶场都必须处于下面四态之一：

1. **已部署未开放**
2. **已开放可复测**
3. **复测中**
4. **复测完成待销毁**

建议进一步扩展为机器可读状态：

- `deployed-closed`
- `open-ready`
- `in-validation`
- `finished-pending-destroy`
- `destroyed`
- `failed`

### 状态转换约束

- `deployed-closed -> open-ready`
  - 必须经过健康检查
  - 必须有端口映射记录
  - 必须明确开放窗口

- `open-ready -> in-validation`
  - 可由审核开始使用时切换
  - 也可以继续保持 `open-ready`，视自动化颗粒度而定

- `in-validation -> finished-pending-destroy`
  - 一旦复测完成，必须尽快进入回收决策态

- `finished-pending-destroy -> destroyed`
  - 默认推荐终态

第一版不建议长期保留大量 `finished-pending-destroy` 环境。

## 七、端口暴露模型（第一版）

## 7.1 总体策略

第一版接受 **直接开放容器端口**，但必须加四条纪律：

### 纪律 A：一个环境一套固定映射规则

每个环境必须明确记录：

- `env_id`
- 服务名
- 容器内端口
- 宿主机端口
- 对应访问 URL
- 是否公网开放
- 当前白名单状态

### 纪律 B：端口必须可追踪

必须维护一份 **端口租约表**，至少记录：

- 端口号
- 归属环境
- 分配时间
- 释放时间
- 当前状态（reserved / open / closed / released）

### 纪律 C：默认不开，按需开

环境拉起不等于公网开放。

建议拆成两层：

1. 容器/宿主机端口映射存在
2. 腾讯云安全组或宿主机防火墙真正放行

只有第二层放行时，才算“已开放”。

### 纪律 D：优先由腾讯云安全组控制公网入口

因为你明确需要在关键时刻限制指定 IP 段，所以第一版建议：

- 以 **腾讯云安全组** 作为第一层公网控制
- 宿主机防火墙可作为补充
- 容器自身不要承担主访问控制职责

---

## 7.2 端口分配建议

第一版建议预留一个可管理区间，例如：

- `20000-20999` 作为 Web/HTTP 靶场端口池
- 其他类型（如特殊 TCP 服务）另设区间

不建议：

- 无规则临时使用任意高位端口
- 端口与项目关系只存在于聊天记录里

---

## 7.3 白名单策略

建议支持三种模式：

1. `closed`
   - 不允许公网访问
2. `open-all`
   - 对所有来源开放（仅在短时间窗口内使用）
3. `allowlist`
   - 只允许指定 IP / IP 段访问

第一版不一定要自动化到极致，但在状态与流程中必须把这三种模式纳入设计。

## 八、环境生命周期与重置策略

## 8.1 第一版默认策略

### 默认策略：**每次正式复测前，重置到干净态**

这是已确认的第一版原则。

原因：

- 避免历史脏数据影响审核结论
- 避免环境被前一次测试改坏
- 避免“我这里能复测、审核那边不行”

## 8.2 重置方式分层

建议支持三层重置方式：

### 方式 A：轻量重置

- 仅重启容器
- 清临时日志 / 缓存
- 不改 DB / volume

适合：排查类短操作，不适合作为正式复测前保证

### 方式 B：环境级重置（推荐正式复测前默认）

- 恢复 DB dump
- 恢复初始 volume
- 重新启动容器

适合：大多数 Web 靶场

### 方式 C：整环境重建

- 删除容器 / 网络 / volume
- 从工作副本或模板重新起环境

适合：

- 环境已明显污染
- 上次复测把现场改坏
- 快照恢复不可信

第一版建议：**把方式 B 作为首选正式重置能力，把方式 C 作为兜底能力。**

## 九、资源预算与并发纪律

## 9.1 第一版资源纪律

已确认：8G RAM 不可变。

因此第一版建议写死以下规则：

1. 默认同时只允许 **1 个重环境**
2. 轻环境可酌情 **2 个**
3. 任意环境构建 / 启动后，都记录一份峰值：
   - CPU
   - RSS / 内存占用
   - 磁盘增长
   - 容器日志量

## 9.2 环境分级建议

建议未来按经验把靶场粗分三类：

### `light`

- 单容器或少量容器
- 不依赖重型构建
- 启动后常驻占用低

### `medium`

- 标准 Web + DB + cache
- 有一定初始化成本
- 但仍可控

### `heavy`

- 多服务、多数据库、构建链重
- 启动和稳定运行都吃资源
- 更适合“单独窗口使用，测完就拆”

第一版不需要复杂调度器，但需要这个概念，为后续 admission rule 打基础。

## 十、交付给审核员的统一模板

未来建议每次对外提供复测环境时，都生成统一交付卡片。

最小模板建议包括：

1. **目标地址**
   - IP:Port
   - 如有多个入口，明确主入口

2. **开放时间窗口**
   - 例如：今天 14:00 - 18:00

3. **PoC 步骤**
   - 尽量按 1/2/3 步给出

4. **测试账号密码**（如有）
   - admin / test / demo 等

5. **注意事项**
   - 是否需登录
   - 是否需先访问某个初始化页面
   - 是否存在验证码 / CSRF / 上传路径限制等

6. **预期结果**
   - 成功时应看到什么
   - 失败时通常表现为何

7. **故障联络 / 续期开窗说明**
   - 如果打不开怎么办
   - 如果要延期复测该如何联系

这个模板不一定要第一天就完全自动化，但必须纳入方案目标。

## 十一、清理与回收策略

## 11.1 总原则

把“可重建”当成默认能力，而不是例外能力。

## 11.2 建议清理对象

第一版至少应考虑：

- 已停止容器
- 长时间不用的 volume
- 悬空镜像
- 无用构建缓存
- 过大的日志文件
- 过期导出文件

## 11.3 回收决策

每个环境复测结束后，必须做一次二选一或三选一：

1. **保留快照后销毁容器**
2. **短期保留环境**（有明确保留截止时间）
3. **立即完全销毁**

不建议默认长期保留运行态。

## 十二、第一版实施阶段划分

## 阶段 A：云机靶场宿主机打底

目标：把“远程可控宿主机”先打稳。

### A 阶段范围

1. SSH 远程控制打通
2. Docker / Compose 可用
3. 宿主机最小防护
4. 腾讯云安全组规则整理
5. 基础目录结构建立
6. 日志 / 清理 / 端口记录方式定下来

### A 阶段产物

- 云机初始化说明
- SSH 驱动约定
- 靶场目录结构
- 端口租约表格式
- 状态文件格式
- 安全组使用约定

## 阶段 B：跑通 1 个完整靶场闭环

目标：先验证单靶场闭环，不急着泛化。

### B 阶段范围

1. MacMini 远程起环境
2. 云机开放端口
3. 外网可访问
4. 按 PoC 成功复测
5. 回收环境
6. 必要时重新拉起

### B 阶段验收标准

- 环境可从 MacMini 一键或近一键拉起
- 对外地址稳定可访问
- 审核员拿说明即可复测
- 复测结束后可关闭/清理
- 能再次恢复到干净态

## 阶段 C：抽象成多靶场运营规则

只有 B 阶段验证成功后再进入。

### C 阶段范围

1. 端口租约表
2. IP 白名单切换
3. 并发上限
4. 自动清理
5. 环境模板化

## 十三、第一版显式确认的 4 条执行策略

以下四点已由用户确认，视为第一版硬约束：

1. **第一版只支持单靶场在线**：是
2. **每次正式复测前重置到干净态**：是
3. **公网开放默认关闭，复测前再开**：是
4. **第一版不做复杂平台化，只做 SSH 驱动远程靶场**：是

这四条应优先于“做得更炫”“做得更自动”“做得更像平台”的冲动。

## 十四、第一版不建议做的事情

为了避免前期跑偏，第一版明确不建议：

1. 不先做完整 Web 管理后台
2. 不先做复杂反向代理平台
3. 不先做大量常驻环境并发
4. 不先做花哨的多节点调度系统
5. 不把云机变成第二个控制面
6. 不把“直接开放容器端口”误升级成“所有端口长期裸开”

## 十五、主要风险与应对

## 风险 1：公网扫描污染复测现场

### 表现

- 默认口令被别人抢先改
- 数据被写脏
- 漏洞被第三方先打

### 应对

- 默认关闭公网
- 正式复测前重置到干净态
- 关键窗口改为 allowlist
- 复测结束后尽快关闭暴露

## 风险 2：云机逐渐雪花化

### 表现

- 哪个环境怎么起来的没人说得清
- 同一个环境只能在“这台脏机器”上复现
- 手工改动越来越多

### 应对

- 配置模板化
- 目录统一
- 运行态与长期状态分离
- 把可贵经验写回 MacMini 侧计划/文档/脚本

## 风险 3：8G 内存下重环境互相踩踏

### 表现

- OOM
- Docker 构建失败
- 多环境一起跑导致整机抖动

### 应对

- 第一版单靶场在线
- 环境分级
- 启动后记录峰值
- 重环境测完即拆

## 风险 4：端口管理失控

### 表现

- 端口冲突
- 忘记哪个端口对外开着
- 审核员拿到过期地址

### 应对

- 端口租约表
- 环境状态机
- 默认关闭公网
- 开窗必须带截止时间

---

## 十六、对当前方案的建议结论

### 结论 1：方向正确

与“把 Loop9 主控整体搬上云”相比，当前方案明显更合理：

- 风险更小
- 结构更清晰
- 更符合当前资源现实

### 结论 2：第一版应该故意做薄

最合适的第一版，不是平台，而是：

- 一台 SSH 可控的 Docker 靶场宿主机
- 一套明确状态机
- 一套端口与开放窗口纪律
- 一条单靶场闭环验证路径

### 结论 3：真正重要的是“运营纪律”而不是“技术花样”

这个项目后期是否稳定，不主要取决于用了多少花哨组件，而取决于有没有把下面这些纪律落实：

- 默认关、按需开
- 正式复测前重置
- 单靶场先跑稳
- 端口可追踪
- 复测后立刻回收决策
