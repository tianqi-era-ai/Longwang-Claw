## 当前初判：优先关注的案例类型
1. **上传链到 RCE**
   - unrestricted upload
   - 路径穿越 + 上传落点控制
   - 远程图片/文件拉取 → 服务端解析 → 写壳 / 命令执行
2. **模板/解释器/表达式执行到 RCE**
   - SSTI / EL / Velocity / Jinja / Handlebars / Mustache 等
   - 适合扩展成跨语言通用方法论
3. **已登录低权限 / 边界功能链到 RCE**
   - 后台弱权限功能、插件、导入、测试连接、任务执行器、hook/webhook
4. **未授权 / SSRF / 文件读写 / 配置控制 组合后通向 RCE**
   - 本轮不做未授权大全，但要保留与 RCE 链接的分析位

## 当前初判：设计上要避免的偏题
- 不要先做成“海量关键词库驱动”的重系统
- 不要只做 Java 专用视角；应先抽象为跨语言/通用 exploitation surface
- 不要把 skill 写成“发现即定性”的黑盒；要留下共享 context，给后续人/流程接着分析

## 已完成的首轮外查（2026-03-18 晚）
- `Attacking HelpDesks Part 1: RCE Chain on DeskPro, with Bitdefender as a Case Study`
  - 价值：非常贴合“未授权/低权限 → 配置/身份 → 模板/反序列化 → RCE”的链式思维
  - 可迁移点：
    - 不要把 RCE skill 只理解成 sink 关键词扫描器；它应该同时盯 **身份面、配置面、模板面、后台功能面**
    - 对 helpdesk / admin panel / CRM 这类系统，要优先关注：自注册、普通用户 token、API 泄露、插件/用户源/SSO/JWT 配置、模板编辑器
- `Magento – RCE & Local File Read with low privilege admin rights`
  - 价值：非常适合抽象出“低权限后台 + 上传落点/路径穿越 + 模板/渲染器执行”方法论
  - 可迁移点：
    - 产品/模板/邮件/主题/设计器 这类 **富配置功能** 是高价值 exploitation surface
    - 文件上传不只看“是否能传 shell”，还要看 **上传后落点可预测性、解释器调用路径、路径拼接/遍历、渲染入口**
- `Unrestricted File Upload to RCE | Bug Bounty POC`
  - 价值：虽然较传统，但对“前端限制与后端真实落盘规则不一致”的提醒很有用
  - 可迁移点：
    - 上传分析必须分层记录：前端限制、预签名/预上传请求、最终落盘路径、MIME/扩展名/内容三者是否分离校验、替代扩展名是否可执行
- `Microsoft Apache Solr RCE Velocity Template`
  - 价值：提醒 skill 不要只盯业务代码；**中间件/控制台/模板引擎/默认管理面暴露** 也是高优先级入口
  - 可迁移点：
    - 对 Java 生态尤其要关注：模板引擎、管理控制台、动态脚本/表达式、可改配置项是否能把“只读功能”推进成执行面

## 当前方法论初判（第一版）
1. **把 RCE 候选分成“执行面”与“推进面”两层**
   - 执行面：命令执行、模板执行、反序列化、脚本解释、动态类加载/反射、插件执行器
   - 推进面：上传、路径遍历、弱鉴权、配置改写、JWT/SSO/插件配置、任务计划、Webhook/Test Connection、邮件模板/主题模板
2. **这个 skill 的价值不应只是报 sink，而应输出“可能成链”的专题上下文**
3. **对微步偏好的适配方向应明确偏向**：
   - 文件上传 → 落盘 → 可访问/可解释 → RCE
   - 未授权/低权限 → 配置改写/模板编辑/后台动作 → RCE
   - 中间件管理面/控制台暴露 → 模板/脚本执行 → RCE
4. **未授权本轮不全面展开，但必须给后续流程留接口**
   - 需要在共享 context 里保留：入口权限、可达角色、是否自注册、是否存在匿名/弱角色调用面、是否发现鉴权边界异常

## 新增架构想法（2026-03-18 晚）
- 用户提出的点子已采纳为当前优先方案：
  - 不直接长期维护两份主模板里的重复增强段落
  - 而是把“RCE / 文件上传 / 越权 等可插拔模块提示块”抽到独立文件 C
  - 由 `loop9_authorized_review.py` 在生成最终 prompt 时，读取这些模块文件并动态拼接到原模板后面或指定位置
- 这条路线的优势：
  - 避免双模板重复修改与同步漂移
  - 更符合未来并列 skill 的可插拔接入方向
  - 更容易在 Python 层维护一个模块文件路径列表，按策略动态拼接
- 在真正改代码前，先完成最后一轮自检：
  - 拼接位置是否足够稳定
  - 是否需要“追加到模板尾部”还是“插入到固定锚点”
  - 若未来有多个模块并存，顺序与冲突风险如何控制
