# 龙王小龙虾｜定时任务与长任务调度说明

这份文档只讲当前公开仓里和“持续推进”直接相关的主线资产，不讲运行态数据。

## 一、目标

龙王小龙虾不是“一次性脚本跑完就结束”的形态。

它当前更接近三层协作：

1. `cron / agentTurn` 负责按节奏触发不同 lane
2. `dispatcher + launch guard` 负责薄门禁、补位、并发和资源保护
3. `heartbeat + status` 负责长任务观察、节流和对外同步

也就是说：
- 调度层负责“什么时候该跑”
- workflow 层负责“具体怎么跑”
- heartbeat 层负责“跑起来以后怎么观察和汇报”

## 二、当前主线文件

### 1. 薄调度与并发门禁

- `workspace/bin/loop9-dispatch`
- `workspace/bin/loop9-launch-guard`
- `workspace/config/loop9-dispatch.json`
- `workspace/plans/loop9-hourly-thin-orchestration-v1.md`
- `workspace/plans/2026-03-26-loop9-concurrency-config-and-launch-guard.md`

这层的职责是：
- 根据 lane 判断本轮是否应该跳过或发车
- 读取并发上限
- 做 launch 前资源检查
- 维护最小锁 / cooldown / defer

它不是新的厚总控平台，只是当前工作流前面的薄门禁。

### 2. cron 执行稿

- `workspace/plans/loop9_cron/loop9-cron-agentturn-audit.md`
- `workspace/plans/loop9_cron/loop9-cron-agentturn-poc.md`
- `workspace/plans/loop9_cron/loop9-cron-agentturn-publish.md`
- `workspace/plans/loop9_cron/loop9-cron-agentturn-delivery-publish.md`
- `workspace/plans/loop9_cron/loop9-cron-agentturn-verify-v4-launch.md`

这些文件不是泛泛说明，而是当前各条 cron lane 的真实执行稿。

它们的共同特点是：
- 先读对应 skill
- 再走 `plan / guard / claim / launch`
- 一次只处理很薄的一段工作
- 不把所有业务判断重新塞进本地 Python orchestrator

### 3. 长任务观察与 heartbeat

- `workspace/HEARTBEAT.md`
- `workspace/heartbeat/loop9-status-dispatch.md`
- `workspace/prompts/heartbeat-loop9-status-subagent.md`
- `workspace/skills/loop9-status/SKILL.md`

这一层负责：
- 什么时候触发状态检查
- heartbeat 与 `loop9-status` 的职责边界
- 10 分钟最小触发间隔
- 通过独立子任务读取 `loop9-status`，而不是让 heartbeat 直接重写状态摘要

这里的关键原则是：
- heartbeat 是调度层，不是展示模板层
- `loop9-status` 才是状态解释和用户可读输出的 canonical skill

## 三、当前推荐的调度理解方式

### 1. cron 不直接等于 workflow

当前 cron 更像是“按节奏叫醒某条 lane”，不是把整条工作流写死在 cron 配置里。

真正的业务判断仍然下沉在：
- `plans/loop9_cron/*.md`
- `bin/loop9-dispatch`
- 各自的 skill

### 2. dispatcher 不是厚平台

当前 `loop9-dispatch` 的定位很克制：
- 只负责门禁、候选补位、并发与短时状态
- 不接管 skill 的语义 owner 地位
- 不把发现、验证、报告全部重写成厚调度器

### 3. heartbeat 不是第二套执行主线

heartbeat 负责的是：
- 观察
- 节流
- 汇报

它不负责：
- 重新实现审计
- 重新实现 real-poc
- 直接拼装 `loop9-status` 的用户展示格式

## 四、当前公开仓里故意不带的调度运行态

这版公开仓不带下面这些 live state：

- 实际 `cron/jobs.json`
- 实际 announce channel / topic 配置
- `workspace/memory/heartbeat-state.json`
- 各类 `automation-state/` 运行记录
- session / logs / delivery queue / devices / credentials

原因不是这些东西不重要，而是它们属于：
- 运行态
- 账号态
- 现场状态

当前公开仓保留的是：
- 真实执行稿
- 真实门禁脚本
- 真实 heartbeat 入口
- 给 AI 还原这套结构所需的静态说明

## 五、如果要把这套东西装回本地

优先顺序建议是：

1. 先恢复 `workspace/bin/`、`workspace/config/`、`workspace/plans/`
2. 再恢复 `workspace/skills/`
3. 再恢复 `workspace/HEARTBEAT.md`、`workspace/heartbeat/`、`workspace/prompts/`
4. 最后根据你自己的 OpenClaw 运行环境补 live cron / heartbeat state / 外部 delivery 配置

也就是说：
- 公开仓负责给出真实骨架
- live schedule / channel / memory state 仍由使用者自己装配

## 六、当前一句话结论

> 龙王小龙虾当前的持续推进机制，不是一个巨型总控平台，而是 `cron/agentTurn -> dispatcher/guard -> skill/workflow -> heartbeat/status` 这条薄分层主线。
