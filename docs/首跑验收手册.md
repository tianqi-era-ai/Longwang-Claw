# LongWangClaw｜首跑验收手册

这份手册回答一个问题：外部用户拿到仓库后，怎样判断它不是只“装上了文件”，而是具备从 Audit 到报告上传的起跑条件。

首跑分两条路径：

- BYO 授权目标：真实端到端能力验收。
- Contract fixture：最小合成契约检查，只验证文件形状和计划解析。

## 0. 装配与本机配置

先回填公开工作区：

```bash
python3 scripts/bootstrap_openclaw_layout.py --workspace-root ~/.openclaw/workspace
```

准备本机私有配置：

```bash
cp workspace/config/longwang.example.json workspace/config/longwang.local.json
```

填完 `workspace/config/longwang.local.json` 后先 dry-run：

```bash
python3 scripts/render_longwang_config.py --profile workspace/config/longwang.local.json --dry-run
python3 scripts/longwang_doctor.py --profile workspace/config/longwang.local.json --json
```

确认 doctor 只剩符合预期的本机差异后，再写入真实配置：

```bash
python3 scripts/render_longwang_config.py --profile workspace/config/longwang.local.json --write
```

## 1. Contract fixture 检查

没有授权目标时，只能跑契约夹具：

```bash
python3 scripts/longwang_doctor.py \
  --profile workspace/config/longwang.example.json \
  --contract-fixture fixtures/contract \
  --json
```

这个检查应该能证明：

- `fixtures/contract/audit-run/` 能被静态发布计划识别。
- `fixtures/contract/report/` 能被标准报告发布计划识别。
- doctor 的 `contract_fixture` layer 为 ready。
- 不需要真实 Feishu、Docker、OpenCode、Codex 或 API key。

这个检查不能证明：

- 真实漏洞发现能力。
- 真实 PoC 复现能力。
- 真实远程 runtime 可用性。
- 真实 Feishu 上传成功。

## 2. BYO 授权目标真实首跑

真实验收必须使用你有授权审计的目标源码。不要把第三方 targets 提交到本仓。

建议目标条件：

- 本地路径可访问，例如 `~/.openclaw/workspace/targets/<repo>`。
- 你有明确授权。
- 依赖相对清晰，能在 local Docker 或远程 Docker/CVM 中尝试搭建。
- 不使用 deliberately vulnerable training ground 作为默认宣传证明。

## 3. 六段验收顺序

### 3.1 Audit

入口：

```bash
cd ~/.openclaw/workspace/Super8
./.opencode/_scripts/loop9_authorized_review.py --policy internal-full-audit <local-target-path>
```

ready 判据：

- wrapper 成功启动。
- `Super8/temp/loop9/` 或 `Super8/temp/loop9-observe/` 出现对应 run/observe 记录。
- 至少有 audit solution / validation 类输出。

blocked 判据：

- `opencode` 或 `tmux` 不存在。
- 目标路径不是本地授权源码。
- wrapper 无法生成任务或无法启动。

### 3.2 Real-PoC 静态编写

入口：

```bash
python3 ~/.openclaw/workspace/skills/loop9-real-poc/scripts/run_loop9_real_poc.py <completed-loop9-run-dir>
```

ready 判据：

- `<loop9-run-dir>/real_pocs/manifest.json` 存在。
- `real_poc_final_status.json` 存在。
- 每个可交付 finding 至少有明确的 PoC 文件状态。

blocked 判据：

- audit run 未完成。
- finding 证据不足，无法生成诚实的 PoC。
- OpenCode agent 未按 profile 渲染。

### 3.3 静态 Audit/PoC 产物上传

入口：

```bash
python3 ~/.openclaw/workspace/skills/loop9-feishu-publisher/scripts/build_publish_plan.py <loop9-run-or-poc-path> --pretty
```

ready 判据：

- publish plan 能枚举 raw audit、validation、Real-PoC 产物。
- Feishu app / wiki / memory 配置已填。

blocked 判据：

- `feishu.enabled=false`。
- appId/appSecret/wiki 配置缺失。
- 输入路径不是可识别的 Loop9 run 或 PoC bundle。

### 3.4 动态环境搭建与 PoC 复现

入口：

```bash
~/.openclaw/workspace/bin/loop9-verify-v4-auto-run.sh
```

ready 判据：

- `dynamic_verify` lane ready。
- `remoteRuntime.kind` 已选择 `local-docker` 或 `tencent-cvm` / `ssh-docker`。
- 对应 Docker/runtime 配置已填。
- verify-v4 产出 round root、stage handoff/verdict、attempt receipt。

blocked 判据：

- runtime 未启用。
- Docker 不可用。
- SSH/CVM 配置缺失且选择了远程 runtime。
- 真实目标无法搭建或缺少运行前提。

### 3.5 本地标准交付报告

入口：

```bash
python3 ~/.openclaw/workspace/skills/loop9-delivery-reports/scripts/build_repo_delivery_reports.py --round-root <reports/.../rounds/<repo>/<round>>
```

或：

```bash
python3 ~/.openclaw/workspace/skills/loop9-delivery-reports/scripts/build_repo_delivery_reports.py --delivery-input-json <delivery-report-input.json>
```

ready 判据：

- `workspace/reports/<repo>/98-delivery-bundle.manifest.json` 存在。
- `99-最终本地复盘.*` 存在。
- 单 finding 报告、PoC、HTTP 证据路径与 manifest 一致。

blocked 判据：

- verify 阶段没有冻结 repo-level truth。
- delivery input 缺少 finding、PoC 或 HTTP 证据绑定。
- 生成结果明显薄于标准报告契约。

### 3.6 动态复现后的标准报告上传

入口：

```bash
python3 ~/.openclaw/workspace/skills/loop9-feishu-delivery-publisher/scripts/build_report_publish_plan.py ~/.openclaw/workspace/reports/<repo> --pretty
```

ready 判据：

- delivery publish plan 能枚举标准报告树。
- `publish_ready=true`。
- Feishu delivery node 配置已填。

blocked 判据：

- 标准报告 manifest 缺失或 kind 不正确。
- Feishu 配置缺失。
- 本地报告树还未完成 final local review。

## 4. 验收输出怎么读

用 doctor 的 JSON 给另一个 AI 继续修补：

```bash
python3 scripts/longwang_doctor.py --profile workspace/config/longwang.local.json --json
```

先看 `layers`：

- `repo_static`：仓库公开资产和边界。
- `local_config`：本机 profile 是否填好。
- `rendered_config`：模板能否渲染。
- `toolchain`：本机命令和语法。
- `contract_fixture`：合成契约夹具。
- `lane_readiness`：六段链路聚合状态。

再看 `lanes`，按 blocked 的 lane 逐条修。不要因为缺 `reports/`、`runs/`、`targets/` 就把运行产物补进公开仓。
