# 龙王小龙虾｜开源范围与仓库关系说明（第一版）

> 说明：这是一版用于统一理解的骨架稿。
> 最终对外仓库命名、主仓/副仓划分、公开节奏，仍需大黑客手动拍板。

## 一、当前已知的两个核心仓库

当前“龙王小龙虾”相关实现，至少涉及两个核心代码位置：

1. `~/.openclaw`
2. `~/.openclaw/workspace/Super8`

这两者不是简单重复关系，更像是 **不同层级的分工**。

---

## 二、建议的理解方式

### 1. `~/.openclaw` 更像控制面 / 宿主层
它承载的更像是：
- OpenClaw 本体与运行环境
- agent / session / channel / tool / skill 等控制面能力
- cron / heartbeat / gateway / delivery queue 等运行机制
- 对外交互入口与长任务承载基础设施

如果从产品叙事来讲，这一层更接近：

> 龙王小龙虾所依附、所调用、所扩展的 AI 控制面与运行底座。

### 2. `~/.openclaw/workspace/Super8` 更像工作流 / 执行主线层
它承载的更像是：
- Loop9 相关长任务与工作流主线
- 审计、环境、PoC、Real-PoC 等执行逻辑
- prompt、observe、报告、workflow 资产
- 更贴近“具体安全工作如何被组织和推进”的实现层

如果从产品叙事来讲，这一层更接近：

> 龙王小龙虾真正把安全工作流跑起来的核心工作台 / 执行主线层。

---

## 三、为什么不是一个仓库全塞完

从当前形态看，这两个仓库分开的价值在于：

### 1. 控制面与执行面不是一回事
- `~/.openclaw` 更偏“系统如何承载任务”
- `Super8` 更偏“任务本身如何被推进”

### 2. 对外受众也不同
- 如果对方关心的是 OpenClaw、Agent、Skill、Tool、channel、cron、session 这些机制，关注点更偏 `~/.openclaw`
- 如果对方关心的是 Loop9、安全工作流、审计闭环、PoC 复现、报告交付，关注点更偏 `Super8`

### 3. 这样更适合做分层讲解
可以把产品叙事分成：
- 上层：AI 原生控制面
- 中层：工作流与调度机制
- 下层：安全任务执行与交付闭环

---

## 四、当前建议的对外讲法

### 稳妥讲法

> 龙王小龙虾当前由两部分核心资产共同支撑：
> 一部分负责 AI 控制面、工具调用、长任务承载与调度；
> 另一部分负责安全工作流本身的执行主线，包括审计、环境、PoC、证据与交付闭环。

### 更产品化一点的讲法

> 一个仓库更像“控制面与运行底座”，
> 另一个仓库更像“安全工作流引擎与执行主线”。

## 五、当前不建议直接写死的点

在你最终拍板前，暂时不建议直接对外写死：

- 哪个仓库一定是主仓，哪个一定是副仓
- 两者将来是否都公开
- 是否会拆成更多公开仓
- 是否把 `Super8` 作为最终对外品牌名之一

这些都属于产品与发布策略，不适合我先替你定死。
