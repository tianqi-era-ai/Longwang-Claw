# 龙王小龙虾｜Skill / Tool / Wrapper / Harness 工程说明（第一版）

## 一、先给结论

龙王小龙虾不是一个“只有一个总程序”的系统，
而是由多层能力共同组成的。

其中最容易混淆、但又必须分清的四层是：

- **Tool**
- **Skill**
- **Wrapper**
- **Harness 工程**

它们不是同义词，也不是谁都能替谁。

一句话先概括：

> **Tool 提供原子动作。**
> **Skill 组织任务做法。**
> **Wrapper 负责外层入口与稳定桥接。**
> **Harness 工程负责在复杂模型与复杂工作流场景下，驾驭不确定性与保持系统可控。**

---

## 二、Tool：原子能力层

Tool 更像是系统能直接调用的原子动作。

例如在 OpenClaw 这一层，Tool 可能包括：
- 读文件
- 写文件
- 编辑文件
- 执行命令
- 调消息系统
- 调浏览器
- 调日历 / 飞书 / 文档 / 表格等外部能力

当前公开仓里，像飞书这类外部能力并不只是一份 workflow skill，
它在 OpenClaw 根层还有真实的 channel/plugin 源码承接，例如：

- `extensions/openclaw-lark/`

Tool 的特点是：
- 边界清楚
- 输入输出相对明确
- 更偏“做一个动作”

所以 Tool 负责的是：

> **让系统具备做具体事情的手脚。**

但 Tool 本身一般不负责告诉系统：
- 这件事整体该怎么做
- 当前应该先做哪一步
- 下一轮主线是什么

---

## 三、Skill：任务组织层

Skill 比 Tool 更高一层。

它不只是“能做一个动作”，而是把某类任务的常见做法、边界、注意事项、调用方式组织起来。

在龙王小龙虾这类系统里，Skill 的意义很大，因为很多任务不是单动作，而是：
- 先读什么
- 再判断什么
- 哪些情况该走哪条路
- 哪些材料该优先引用
- 哪些边界不能越

例如你已经明确点名的代表性能力里，就有：
- 自动化环境搭建 Skill
- 自动化 PoC 复现 Skill

这说明 Skill 在龙王小龙虾里，不是装饰层，
而是：

> **把某类复杂任务沉淀成可复用的操作法与工作流单元。**

---

## 四、Wrapper：外层入口与稳定桥接层

Wrapper 不是主线 owner，但它很重要。

它更适合承担的是：
- 对外统一入口
- 路径与参数整理
- prompt / task file 组装
- 把某个稳定调用路径包起来
- 替外部交互层屏蔽底层复杂度

例如前面已有资料里就明确提过一种设计思路：
- Telegram / OpenClaw 侧只做外层入口
- 真正执行统一走稳定 wrapper

这就是典型的 Wrapper 价值。

所以 Wrapper 更像：

> **把外部触发和内部真实执行之间，搭一层稳定、清晰、可维护的桥。**

但这里有一个特别重要的边界：

### Wrapper 不能偷偷篡位成主线 owner
也就是说，Wrapper 不能悄悄变成：
- 真正决定意义的人
- 真正决定 owner / frontier / 下一跳的人
- 真正决定整条主线逻辑的人

否则系统就会从 AI 原生工作流，慢慢滑回“大 wrapper / 大脚本控制面”。

---

## 五、Harness 工程：高阶控制面与不确定性驾驭层

Harness 工程是更高阶的一层，因此你已经拍板把它放在“进阶和高级内容”里。

它之所以重要，不是因为它听起来高级，
而是因为复杂 AI 系统天然会遇到这些问题：
- 模型随机波动
- 偶发降智
- 同样任务有时做得很好，有时不稳
- 长工作流里中途跑偏
- 子步骤看似都存在，但整体难以持续推进

Harness 工程面对的，就是这些问题。

它更接近：
- 让复杂系统可控
- 让不确定性可被驾驭
- 让高随机性的模型行为，尽量被纳入可持续推进的轨道

所以可以把它理解成：

> **龙王小龙虾在复杂模型 + 复杂工作流场景下，为了不失控而长出来的高阶工程层。**

---

## 六、这四层为什么必须分清

如果不分清，系统很容易出现几种混乱：

### 1. 把 Tool 当成工作流
结果就会变成：动作很多，但整体不会推进。

### 2. 把 Skill 写成一堆 Tool 使用说明
结果就会失去“任务组织层”的价值。

### 3. 把 Wrapper 做成真正的总控
结果就会让系统重新滑向脚本主导、程序主导，而不是 AI 原生主线。

### 4. 把 Harness 当成“多写几个脚本”
那就会误解它真正的价值。
Harness 真正关心的不是多几个文件，而是系统在高随机性环境下是否还能稳定承担复杂任务。

---

## 七、在龙王小龙虾里的推荐分工理解

### Tool
负责：
- 原子动作
- 基础调用能力
- 和外部世界接触的“手脚”

### Skill
负责：
- 某类任务的可复用做法
- 工作流单元化
- 操作法沉淀

### Wrapper
负责：
- 外层统一入口
- 稳定桥接
- 降低外部调用复杂度

### Harness 工程
负责：
- 高阶稳定性
- 不确定性驾驭
- 防止复杂模型 + 复杂工作流失控

这四层一起，才能让龙王小龙虾既：
- 能调用真实能力
- 能复用任务做法
- 能有稳定入口
- 又能在高复杂场景里不轻易崩盘

---

## 八、这和“万能工作流”有什么关系

你已经明确把“万能工作流”视为代表性成果之一。

而“万能工作流”之所以能成立，恰恰依赖这几层不要混乱：
- Tool 不能替代 Skill
- Wrapper 不能替代主线 owner
- Harness 不能被误当成普通脚本集合

只有分层清楚，Loop9 这类工作流才更可能：
- 不只服务当前安全场景
- 还能迁移到其他高难度任务里

也就是说：

> 这四层分工清楚，本身就是“万能工作流”能够成立的一个重要条件。

---

## 九、推荐的对外讲法

### 稳妥版
龙王小龙虾不是把所有逻辑塞进一个总程序里，而是通过 Tool、Skill、Wrapper、Harness 等不同层级协同，来组织复杂工作流。

### 更明确一点的版本
Tool 解决“能做什么”，Skill 解决“这类任务通常该怎么做”，Wrapper 解决“外部如何稳定触发”，Harness 解决“复杂模型和复杂工作流如何不失控”。

### 更适合传播的一句话
它不是一个大脚本，而是一套分层组织出来的 AI 原生工作流工程。

---

## 十、当前一句话总结

> 在龙王小龙虾体系里，**Tool、Skill、Wrapper、Harness** 分别承担原子动作、任务组织、外层稳定桥接、高阶不确定性驾驭这四种不同职责；正是这套分层，让它不只是“能跑几个动作”，而更像一套能长期推进复杂任务的 AI 原生工作流工程。
