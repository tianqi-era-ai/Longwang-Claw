# LongWangClaw｜闭环契约地图

这份地图把 LongWangClaw 的六段主链路写成可检查契约。它用于装配、doctor、首跑和后续 AI 修补。

## 总链路

```text
Audit
  -> Real-PoC 静态编写
  -> 静态 Audit/PoC 产物上传
  -> 动态环境搭建与 PoC 复现
  -> 本地标准交付报告
  -> 动态复现后的标准报告上传
```

## 1. Audit

- owner：`loop9-wrapped-audit` + Super8 OpenCode wrapper。
- 入口：`workspace/Super8/.opencode/_scripts/loop9_authorized_review.py`。
- 必要配置：`paths.workspaceRoot`、`paths.super8Root`、`commands.opencode`、`commands.tmux`、`models.opencode.agentModel`。
- 输入：本地授权目标路径，或用户明确授权后物化到本机的目标路径。
- 输出：`Super8/temp/loop9/...` 与 observe/status 产物。
- ready：wrapper、OpenCode command、agents、prompt 模板存在，本机命令可用。
- blocked：目标未授权、目标不是本地路径、OpenCode/tmux 缺失、wrapper 缺失。
- 下游：Real-PoC 读取 completed Loop9 run。

## 2. Real-PoC 静态编写

- owner：`loop9-real-poc`。
- 入口：`workspace/skills/loop9-real-poc/scripts/run_loop9_real_poc.py <completed-loop9-run-dir>`。
- 必要配置：Audit run 可读，OpenCode command/agents 可用。
- 输入：已完成的 Loop9 audit run。
- 输出：`<run>/real_pocs/manifest.json`、`real_poc_final_status.json`、PoC 文件、round solution/validation。
- ready：PoC manifest 和 final status 能描述 finding 到文件的映射。
- blocked：audit run 未完成、finding 证据不足、PoC 文件缺 source trace 或状态不诚实。
- 下游：静态产物上传读取 audit run 和 `real_pocs/`。

## 3. 静态 Audit/PoC 产物上传

- owner：`loop9-feishu-publisher` + `extensions/openclaw-lark`。
- 入口：`workspace/skills/loop9-feishu-publisher/scripts/build_publish_plan.py <path>`。
- 必要配置：`feishu.enabled`、`feishu.appId`、`feishu.appSecret`、wiki 配置、`paths.memoryRoot`。
- 输入：Loop9 run 或 PoC bundle。
- 输出：Feishu publish plan；真实上传状态写入本机 memory。
- ready：publish plan 能枚举原始输入、审计过程、审计结论、验证报告、PoC 与验证。
- blocked：Feishu 配置缺失、输入形状无法识别、插件缺失。
- 下游：raw artifacts 已进入协作/知识库侧，但不等于标准交付报告。

## 4. 动态环境搭建与 PoC 复现

- owner：`loop9-verify-v4` parent skill；env-bootstrap/finding-replay/distillation 是 bounded child stage。
- 入口：`workspace/bin/loop9-verify-v4-auto-run.sh` 或显式读取 `workspace/skills/loop9-verify-v4/SKILL.md`。
- 必要配置：`commands.codex`、`commands.docker`、runtime profile、`paths.runsRoot`、`paths.reportsRoot`。
- 输入：待验证 repo、finding queue、Real-PoC 或 audit truth。
- 输出：round root、stage handoff/verdict、attempt receipt、repo-level closure/readout。
- ready：runtime 选择明确，local Docker 或 SSH Docker/CVM 配置完整，parent 保持 repo-level truth owner。
- blocked：runtime disabled、Docker 不可用、SSH 字段缺失、目标环境不可搭建、finding 缺少必要前提。
- 下游：本地标准报告只消费已冻结的 repo-level truth。

## 5. 本地标准交付报告

- owner：`loop9-delivery-reports`，脚本只做确定性 materialization。
- 入口：`build_repo_delivery_reports.py --round-root ...` 或 `--delivery-input-json ...`。
- 必要配置：`paths.reportsRoot`、`paths.runsRoot`。
- 输入：repo closure truth、delivery-report-input、attempt receipts、PoC 和 HTTP 证据。
- 输出：`workspace/reports/<repo>/` 标准报告树。
- ready：`98-delivery-bundle.manifest.json`、`99-最终本地复盘.*`、单 finding 报告、PoC、HTTP 证据齐全。
- blocked：repo-level truth 未冻结、manifest 缺失、PoC/HTTP 绑定缺失、报告树明显薄化。
- 下游：动态报告上传读取标准报告树，不重新发现 runtime artifacts。

## 6. 动态复现后的标准报告上传

- owner：`loop9-feishu-delivery-publisher` + `extensions/openclaw-lark`。
- 入口：`build_report_publish_plan.py ~/.openclaw/workspace/reports/<repo>`。
- 必要配置：Feishu app、delivery wiki node、memory 路径。
- 输入：标准报告树。
- 输出：标准报告发布计划；真实上传状态写入本机 memory。
- ready：manifest kind 为 `loop9-standard-delivery-report`，`publish_ready=true`，docs 可枚举。
- blocked：报告树不完整、Feishu 配置缺失、delivery node 缺失。
- 下游：交付完成后进入复盘、沉淀和后续运行态清理。

## Runtime 契约

`longwang.local.json` 必须明确选择 runtime：

- `local-docker`：只需要本机 Docker、runtimeRoot、dockerRegistryPath、publicBaseUrl。
- `tencent-cvm` / `ssh-docker`：需要 SSH host/user/key、runtimeRoot、dockerRegistryPath、publicBaseUrl。

doctor 不把 `local-docker` 当作 CVM 检查，也不会把示例 profile 的占位符当作真实配置。

## Contract fixture 契约

`fixtures/contract/` 只检查两个解析面：

- `audit-run/` 能被静态发布计划识别。
- `report/` 能被标准报告发布计划识别。

它不进入 bootstrap，不回填到 targets，不触发真实上传，不声明真实漏洞能力。
