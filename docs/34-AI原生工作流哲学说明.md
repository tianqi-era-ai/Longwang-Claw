# 龙王小龙虾｜AI 原生工作流哲学说明（第一版）

## 一、先给结论

龙王小龙虾的核心，不是“多写一点脚本，把流程串起来”，
而是：

> **让 AI 拥有对复杂任务主线的语义控制权，让代码只负责确定性的硬边界。**

这就是它的 AI 原生工作流哲学。

如果把这句话说得更直接一点：

- **AI 应该拥有“意义”**
- **代码应该只负责“桥接”**

---

## 二、为什么不能走“大 Python orchestrator”路线

复杂 AI 工作流里，一个很常见的错误，是看起来越来越工程化，实际上越来越偏离主线。

最典型的表现是：
- 开始长出很厚的 Python orchestrator
- 开始长出越来越重的 runner / schema / protocol / registry
- 开始让脚本替 AI 去决定：
  - 当前 owner 是谁
  - 当前 frontier 是什么
  - 下一步该往哪走
  - 哪个问题算“同一类问题”

短期看，这样像是在“增强系统”。
但长期看，这会让系统越来越像：
- 代码在掌管意义
- AI 只是在填空
- 工作流越来越僵
- 难题一变形，系统就鬼打墙

所以龙王小龙虾的默认立场不是：
- “先把控制逻辑都程序化”

而是：

> **先让 AI 负责理解当前局势，再让程序负责把确定性的动作稳稳做掉。**

---

## 三、什么叫“AI 拥有意义”

在龙王小龙虾这类系统里，AI 更应该拥有的是这些东西：

### 1. 当前局势理解
比如：
- 现在卡在哪
- 当前主线还在不在
- 这是环境问题、验证问题、还是叙事问题

### 2. frontier 判断
也就是：
- 当前最该推进的前沿是什么
- 下一跳应该落在哪层
- 哪个问题是真 blocker，哪个只是噪音

### 3. pattern 识别
AI 应更擅长识别：
- 这是不是旧问题回跳
- 这是不是同一类结构性卡点
- 当前系统有没有在空转或假进展

### 4. 主线 owner 维护
最关键的一点是：

> 主线的“意义所有权”不该悄悄滑到脚本、runner、schema、registry 手里。

如果主线 owner 不在 AI 这边，系统很容易表面在推进，实际在机械打转。

---

## 四、什么叫“代码只负责桥接”

代码并不是不重要。
相反，它非常重要。
但它应该承担的是 **确定性的硬边界**，而不是语义总控。

更适合交给代码的事情包括：

### 1. 文件读写
- 状态落盘
- 结果保存
- 文档写入
- 路径处理

### 2. 执行入口
- 调某个工具
- 发某个命令
- 调某个 Skill
- 跑某个确定性脚本

### 3. receipt / handoff / verdict 的薄落盘
比如：
- 结果对象写成 JSON / Markdown
- 某次执行的 receipt 被保存下来
- 某次 handoff 被规范化输出

### 4. 安全边界与 fail-close
- 权限边界
- 时间预算
- 回滚保护
- 硬失败时的安全退出

一句话：

> 代码应该负责“把事情做稳”，不应该负责“替 AI 决定事情到底意味着什么”。

---

## 五、为什么这对龙王小龙虾特别重要

龙王小龙虾不是一个简单问答系统。
它面对的是：
- 多阶段工作流
- 长任务推进
- 环境与验证之间的反复切换
- 不确定性很高的复杂问题
- 模型可能出现的随机性和降智

在这种系统里，如果语义控制权滑到程序那边，会有几个典型后果：

### 1. 容易把“结构完整”误当成“真实进展”
比如：
- 写了更多对象
- 多了更多字段
- 状态看起来更丰满

但实际上并没有新证据、没有 receipt、没有真实下一轮行为变化。

### 2. 容易把 request / pending / probing 当成完成
也就是系统明明只是“还在请求、还在探测、还没被消费”，
却假装自己已经进展很大。

### 3. 容易在复杂任务里长出“假的稳”
看起来脚本越来越多、协议越来越全、流程图越来越漂亮，
但一遇到新问题、异形问题、语义变体问题，马上就卡死。

所以龙王小龙虾必须坚持一个方向：

> **真正的稳，不是厚控制面，而是 AI 主线理解 + 薄程序桥接。**

---

## 六、AI 原生系统里，什么才算真实进展

龙王小龙虾这类系统，不应该把“表面有动作”当成真实进展。

真正算进展的，至少要落到这些东西之一：

- 新的被接受证据
- 真实 receipt
- 被消费的 handoff
- 改变下一轮默认行为的 live-used 变化
- 真正改变主线 owner / frontier 的更新

如果没有这些，很多“看起来在做事”的动作，都不应被误判成真正前进。

---

## 七、它如何理解“自进化”

龙王小龙虾当然追求进化，但不是那种伪自进化。

### 伪自进化长这样
- 多了几个状态字段
- 多了几个提示词对象
- 多了几条规则
- 多了一个更大的 orchestrator

这些不自动等于系统真的更强了。

### 真自进化至少要满足
1. 来自真实结果与真实问题
2. 改变下一轮默认行为
3. 下一轮真的被使用了
4. 出问题还能回滚

所以它要追求的不是“越长越大”，
而是：

> **让真实经验真正改变下一轮主线行为。**

---

## 八、它如何面对模型随机性与降智

龙王小龙虾并不假装模型永远稳定。
相反，它把这些视为原生现实的一部分：
- 输出波动
- 随机降智
- 同一件事有时做得好、有时做得差
- 复杂任务中途跑偏

这意味着它不能把：
- 一次成功，当成永远鲁棒
- 一次失败，当成系统彻底错误
- timeout，当成 prompt 必然有问题

更合理的做法是：
- 把 retry / replay / compare 视为 dispatcher-native 的责任
- 把 timeout 先看成调用预算问题，而不是语义设计自动有罪
- 把“随机波动”和“结构性设计失败”分开判断

而 Harness 工程之所以重要，也正是在这里：

> 它帮助系统在复杂模型和复杂工作流场景里，更稳地驾驭这些不确定性。

---

## 九、为什么这套哲学和龙王小龙虾的开源价值有关

如果龙王小龙虾只是一些安全脚本的集合，那它当然也能开源。
但那样的开源价值有限。

真正更大的价值在于：

### 1. 它开源的不是“脚本堆”，而是一套 AI 原生工作流思想
这会对：
- AI workflow
- agent 系统
- 安全工作流
- OpenClaw 上层产品范式

都更有参考价值。

### 2. 它提供了一条不同于“厚程序控制面”的路线
很多人做复杂 AI 系统，很容易重新长成：
- 大 orchestrator
- 大 registry
- 大 protocol
- 大脚本中心

龙王小龙虾要展示的是另一条路：

> **AI 拥有主线意义，代码守住确定性边界。**

### 3. 它让“万能工作流”这件事变得更可信
因为只有在这套哲学下，Loop9 这类工作流才更可能从“安全特例”长成“可迁移到其他高难度任务的通用成果”。

---

## 十、当前一句话总结

> 龙王小龙虾的 AI 原生工作流哲学是：**让 AI 拥有复杂任务主线的语义控制权，让代码只负责确定性的硬边界与薄桥接**；它反对用越来越厚的 Python 控制面去假装稳定，而更强调真实前沿判断、薄 handoff / receipt / verdict、以及对随机性与降智的正面驾驭。
